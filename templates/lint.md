扫描 wiki/ 目录中的所有 .md 文件（INDEX.md、_compile-log.md、_changelog.md 除外），逐一检查以下问题并修复：

## 检查项

### 0. 文件夹结构初始化（首次 lint 时的特殊处理）

如果发现 wiki/ 下没有任何子文件夹，或者大量文章直接散落在 wiki/ 根目录：

1. 读取 INDEX.md 中的分类结构（如果 INDEX.md 不存在或没有分类，先扫描所有文章内容，自动生成分类）
2. 为每个分类创建对应的子文件夹
3. 根据每篇文章的内容和 type，将其移动到最匹配的分类文件夹中
4. **全局更新所有 `[[]]` 链接**，确保链接不会因为文件移动而断裂
5. 更新 INDEX.md，确保分类和文件夹一一对应
6. 向用户报告：创建了哪些文件夹、每个文件夹放了哪些文章

这个步骤只在文件夹结构缺失时触发。如果文件夹已经存在，跳过此步骤，进入常规检查。

### 1. Frontmatter 检查
- 文件是否以 `---` 开头的 YAML frontmatter 块开始？
- frontmatter 是否包含所有必需字段：topic, type, status, freshness, updated, tags, sources？
- type 是否是 CLAUDE.md 中定义的合法值之一？
- status 是否是以下合法值之一：draft, review, stable？
- freshness 是否是以下合法值之一：evergreen, volatile？
- updated 日期格式是否为 YYYY-MM-DD？
- tags 是否为数组格式？
- sources 是否为数组格式且指向 raw/ 中的文件？

**修复方式**：缺失的 frontmatter 根据文章内容自动补全。格式错误的字段修正为正确格式。

### 2. 链接格式检查
- "相关概念"、"相关链接"部分的列表项是否使用了 `[[文件名]]` 格式？
- 正文中提到的其他 wiki 文章对应概念，是否使用了 `[[]]` 链接？
- 是否存在纯文本的概念引用应该改为 `[[]]` 链接？

**修复方式**：将纯文本引用替换为 `[[]]` 链接格式。

### 3. 摘要检查
- frontmatter 之后的第一段是否为 2-3 句话的摘要？
- 摘要是否直接开始（不带标题），概括了文章核心内容？

**修复方式**：如果缺少摘要，根据文章内容生成一段。

### 4. 文件命名检查
- 文件名是否遵循中文优先原则？
- 能用中文表达的普通概念是否错误地使用了全英文命名？
- 专有名词和技术术语保留英文是正确的
- 文件名中是否有空格？（应使用连字符 `-` 连接）
- 判断标准：如果这个词在中文语境中通常直接说中文，文件名就该用中文

**修复方式**：报告命名不规范的文件，给出中文重命名建议，但不自动重命名（因为会影响其他文章的 `[[]]` 链接，需要全局替换）。如果用户确认，再执行重命名并全局更新所有引用该文件的链接。

### 5. 文件夹归属检查
- 文章是否放在了和 INDEX.md 分类对应的子文件夹中？
- 是否有文章直接散落在 wiki/ 根目录（应放入分类文件夹）？
- 文件夹名称是否和 INDEX.md 中的分类名称一致？

**修复方式**：将错误归属的文件移动到正确的分类文件夹，并全局更新所有受影响的 `[[]]` 链接。

## 执行流程

1. 检查 wiki/ 是否有子文件夹结构，如果没有则执行"文件夹结构初始化"
2. 列出 wiki/ 中所有 .md 文件（遍历子文件夹）
3. 逐个检查上述所有项
4. 自动修复可以安全修复的问题（frontmatter 补全、链接格式修正、摘要补充、文件夹归属修正）
5. 对于需要人工确认的问题（文件重命名），只报告不修改
6. 修复完成后，输出一份报告：
   - ✅ 通过检查的文件数量
   - 🔧 自动修复的文件数量和修复内容
   - 📁 文件夹结构变更（如果有）
   - ⚠️ 需要人工处理的问题
   - 📊 整体合规率（通过数 / 总数）
7. 在 `wiki/_changelog.md` 顶部追加本次 lint 记录，包含修复了什么、合规率等信息
